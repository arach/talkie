# Taxonomy prior art: workflows · skills · actions

**Status:** research brief · prior-art only · 2026-05-20  
**Context:** Follow-up to `docs/planning/2026-05-20-workflows-skills-actions-taxonomy.md`  
**Scope:** User-facing automation vocabulary in adjacent tools. This brief intentionally does **not** recommend a Talkie taxonomy.

## Apple Shortcuts

Apple's top-level user-facing artifact is the **Shortcut**. Apple explains an **action** as the building block of a shortcut: a single step in a task, with each shortcut made from one or more actions. The result is a strongly collapsed model: users collect and run shortcuts, while the editor exposes a catalog of actions as the atomic verb tier. The legacy word "workflow" is mostly historical in this product line; Shortcuts replaced the Workflow app, but modern Shortcuts copy centers on shortcuts and actions rather than workflows. Sources: [Shortcuts User Guide](https://support.apple.com/en-gb/guide/shortcuts/welcome/ios).

Shortcuts separates manual and triggered use by surfacing **Automation** as a distinct area, not by renaming the underlying runnable unit. A normal shortcut is run from the app, widget, or share sheet; personal automation uses event/travel/communication/settings triggers to run actions automatically. The authoring seam is still the action-based Shortcuts editor: automations choose a trigger and then reuse the same action vocabulary, rather than introducing a separate advanced workflow product. Sources: [Intro to personal automation in Shortcuts](https://support.apple.com/en-mide/guide/shortcuts/apd690170742/9.0/ios/26).

## Raycast

Raycast is less a single automation canvas and more a launcher with several first-class command-like objects. Its developer terminology says **extensions** add functionality, consist of one or more **commands**, and those commands appear in Raycast's root search. Within a command, **actions** live in the Action Panel as contextual operations such as copying a link or modifying a selected issue. User-created automation shows up in several product-specific buckets: **Quicklinks** for parameterized URLs/deeplinks, **Script Commands** for scripts exposed as searchable commands, and **AI Commands** for reusable prompts. Sources: [Raycast terminology](https://developers.raycast.com/information/terminology), [Quicklinks](https://manual.raycast.com/quicklinks), [Script Commands](https://manual.raycast.com/script-commands), [AI Commands](https://manual.raycast.com/ai/ai-commands).

Raycast is also a notable edge case because it now uses the word **Skills**, but not as the same thing as a command/workflow/action. Raycast Skills are reusable AI context stored as `SKILL.md`; the docs explicitly distinguish Skills from AI Extensions: extensions give the model tools/actions it can take, while skills give it context/instructions. So Raycast does surface command, action, tool, extension, AI command, and skill, but these are not three peer automation objects; they are separated by packaging, invocation surface, contextual action menus, AI tool use, and reusable AI context. Sources: [Raycast Skills](https://manual.raycast.com/ai/skills), [AI Extensions](https://manual.raycast.com/ai/ai-extensions).

## Alfred

Alfred's power-user automation concept is the **Workflow**. In Alfred 5, users create workflows in a visual editor, add **objects** from a palette, and connect them on a canvas. The object palette groups building blocks into **Triggers, Inputs, Actions, Automations, Utilities, and Outputs**. In this vocabulary, "workflow" is the container, "object" is the generic building-block tier, and "action" is one category of object that does work such as launching files, opening URLs, or running scripts. Sources: [Getting Started with Alfred Workflows](https://www.alfredapp.com/help/workflows/getting-started/), [Workflow Editor and Palette](https://www.alfredapp.com/help/workflows/getting-started/editor-and-palette/), [Workflow Actions](https://www.alfredapp.com/help/workflows/actions/).

Alfred separates invocation style inside the workflow/object model rather than creating a separate top-level concept. A workflow can start from keyword inputs, hotkeys, snippets, external triggers, remote triggers, fallback search, and other trigger/input objects. Alfred 5's **Automation Tasks** are positioned as no-code building blocks inside workflows, not as a replacement top-level noun. The authoring seam is between simple built-in features/templates and the full workflow editor with scripts, variables, and connected objects. Sources: [Workflow Triggers](https://www.alfredapp.com/help/workflows/triggers/), [Keyword to Web Search guide](https://www.alfredapp.com/help/workflows/getting-started/keyword-to-web-search/).

## Zapier

Zapier's top-level user-facing artifact is the **Zap**: an automated workflow connecting apps. Zapier defines each Zap as having a **trigger** that starts it and one or more **actions** that run after it triggers. In addition to actions, Zapier exposes adjacent step types such as searches, filters, formatter steps, delays, paths, webhooks, code, and custom/API-request actions, but the public model is still anchored by one branded container noun: Zap. Sources: [What is a Zap?](https://help.zapier.com/hc/en-us/articles/8496309697421-What-is-a-Zap), [Zaps quick start guide](https://help.zapier.com/hc/en-us/articles/22234847450893-Zaps-quick-start-guide).

Zapier mostly treats triggered automation as the default; manual execution appears as testing, run/replay, or special trigger types rather than a separate "manual workflow" object. It does have a constrained-to-advanced seam: users can start from templates or AI/Copilot, then graduate inside the same Zap editor to multi-step Zaps, filters, paths, code, webhooks, custom actions, and API requests. The vocabulary remains Zap/trigger/action/step rather than action/skill/workflow as separate peer concepts. Sources: [Learn key concepts in Zaps](https://help.zapier.com/hc/en-us/articles/8496181725453-Learn-key-concepts-in-Zaps).

## Make (Integromat)

Make's top-level automation artifact is the **Scenario**. Make describes a scenario as an automated workflow and as a series of **modules** that transfer and transform data between apps/services. Its atomic tier is **module**, not step or action, although module types include **triggers, searches, actions, and universal modules**. A trigger module is normally the first module and starts a scenario by watching for or receiving data; action modules process retrieved data and can create, update, delete, pin, save, download, or otherwise operate in services. Sources: [Create your first scenario](https://help.make.com/create-your-first-scenario), [Types of modules](https://help.make.com/types-of-modules).

Make's triggered/manual split is primarily embedded in scenario scheduling and trigger-module type: polling triggers run on a schedule, instant triggers receive webhooks, and scenarios can also be run/tested manually in the builder. Make's authoring surface is already the visual advanced graph; the simplification layer is more about templates, scenario starters, and now AI/agent-facing features than a separate consumer vocabulary. Make also supports **subscenarios**, where one scenario calls another, preserving "scenario" as the reusable automation unit even when modularizing complex flows. Sources: [Subscenarios](https://help.make.com/subscenarios), [Scenario inputs and outputs](https://help.make.com/scenario-inputs-and-outputs).

## n8n

n8n's top-level artifact is the **Workflow**: a collection of **nodes** connected on a workflow canvas to automate a process. The atomic tier is **node**, with node type/function doing the semantic work. n8n documentation distinguishes trigger nodes and action nodes: trigger nodes start workflows and provide initial data, while action nodes perform operations within the workflow. Sources: [Create a workflow](https://docs.n8n.io/workflows/create/), [Nodes](https://docs.n8n.io/workflows/components/nodes/), [Node types: Trigger and Action](https://docs.n8n.io/integrations/creating-nodes/plan/node-types/).

n8n makes manual vs automatic execution a workflow state and trigger-node issue. Workflows can be run manually while building/testing, especially if they have no trigger node; workflows with trigger or webhook nodes must be published to run automatically, and then run when their trigger conditions are met. The escape hatch lives inside the same node canvas through Code nodes, HTTP Request nodes, sub-workflow execution, custom/community nodes, and manual/trigger nodes, not through a separate "skill" layer. Sources: [Create and run workflows](https://docs.n8n.io/workflows/create/).

## IFTTT

IFTTT's top-level artifact is the **Applet**. Applets are automations connecting two or more **services**, and at minimum require a **Trigger** and an **Action**. In developer-facing docs, Applets can also include **Queries** and filter code; triggers and queries provide **ingredients** that can be used to populate action fields. The old "recipe" mental model survives culturally, but current product/docs center on applets, services, triggers, actions, queries, fields, and ingredients. Sources: [What are Applets?](https://help.ifttt.com/hc/en-us/articles/37000197879323-What-are-Applets-and-how-do-I-enable-them), [IFTTT docs](https://ifttt.com/docs), [Glossary](https://ifttt.com/docs/glossary).

IFTTT is strongly trigger-first: applets run automatically when trigger criteria are met after being enabled. Manual authoring is constrained by the "If This Then That" builder, but Pro/Pro+ expands the same applet model with multiple actions, queries, and filter code rather than introducing an advanced graph with a new name. IFTTT therefore presents one memorable container noun, a highly conventional trigger/action split, and optional lower-level developer vocabulary. Sources: [Creating your own Applet](https://help.ifttt.com/hc/en-us/articles/360021401373-Creating-your-own-Applet), [IFTTT docs](https://ifttt.com/docs).

## Tasker

Tasker does not collapse everything into one container noun. Its main screen exposes four main building blocks: **Profiles, Tasks, Scenes, and Variables**. A **Profile** links **Contexts** on the left to **Tasks** on the right; contexts decide when a profile becomes active, and entry/exit tasks run on activation/deactivation. A **Task** is a set of **actions** performed sequentially, and named tasks can be reused from more than one profile or exposed as widgets/shortcuts. Sources: [Tasker main screen](https://tasker.joaoapps.com/userguide/en/activity_main.html), [Tasker tasks](https://tasker.joaoapps.com/userguide-donut/en/tasks.html).

Tasker's triggered/manual distinction is explicit and multi-concept: profiles are context-triggered automations, while tasks can also run from widgets, shortcuts, scene elements, or the task editor's test/run controls. The escape hatch is not a separate "advanced mode" so much as Tasker's native model: contexts, profile properties, task priorities, action groups, variables, scenes, JavaScript, intents, and app creation all sit in the same expert-oriented environment. This is one of the clearer examples where users learn multiple nouns because the product exposes scheduling/triggering and action sequences as separable primitives.

## Keyboard Maestro

Keyboard Maestro's top-level automation unit is the **Macro**, organized into **Macro Groups**. The official wiki frames the system around macro groups, triggers, and actions: macro groups contain macros and control when they are available; **Triggers** describe the event that causes a macro to execute; **Actions** describe what the macro does at each step. Sources: [Keyboard Maestro wiki home](https://wiki.keyboardmaestro.com/Home_Page), [Triggers](https://wiki.keyboardmaestro.com/Triggers).

Keyboard Maestro handles manual vs automatic execution through macro triggers rather than separate object names. A macro may have multiple triggers, ranging from hot keys, typed strings, palettes, status menus, URLs, folders, cron/time, app changes, clipboard changes, USB devices, web server, and more. The advanced seam is additive: users can start with a hotkey macro and later add more triggers, conditions, variables, tokens, scripting, subroutines, and groups without renaming the artifact away from "macro." Sources: [Macro Triggers](https://wiki.keyboardmaestro.com/manual/Macro_Triggers), [Features](https://wiki.keyboardmaestro.com/manual/Features).

## Microsoft Power Automate

Power Automate's top-level unit is the **Flow**, especially **cloud flow** and **desktop flow** in the product vocabulary. Microsoft describes cloud flows as automated workflows that connect apps/services, can be triggered by events or times, and can perform actions across multiple services. The atomic vocabulary is conventional: a **trigger** starts a flow and defines when it runs; **actions** are the events/work the flow does after the trigger. Sources: [Overview of cloud flows](https://learn.microsoft.com/en-us/power-automate/overview-cloud), [Work with triggers and actions](https://learn.microsoft.com/en-au/power-automate/work-with-triggers-actions).

Power Automate makes triggered vs manual a type distinction inside the flow family: **automated cloud flows** are event-triggered, **instant cloud flows** are started by button/control selection, and **scheduled cloud flows** run on a schedule. The authoring seam is increasingly split between Copilot natural-language creation and manual designer creation, with both ending in the same cloud-flow designer; Microsoft also maintains new/classic designers and a separate desktop-flow product for RPA. Sources: [Overview of cloud flows](https://learn.microsoft.com/en-us/power-automate/overview-cloud).

## Apple Automator (legacy macOS adjacent)

Automator's vocabulary is a useful older macOS reference: users create **workflows** by arranging **actions**, and Apple support pages organize the guide around creating/editing workflows, getting information about actions, using results from a previous action, changing action order, quick action workflows, and running workflows. This is the most literal "workflow/action" pairing in Apple's older Mac automation language. Sources: [Automator User Guide](https://support.apple.com/en-gb/guide/automator/ensm2685/mac).

Automator also shows the older macOS pattern for authoring escape hatches: a visual action pipeline can be saved as workflow-like documents and extended with AppleScript, shell script, JavaScript, loops, variables, and imported actions. It does not use "skill" as a user-facing concept; the distinction is between workflow document types and the actions/scripts inside them. Sources: [Automator User Guide](https://support.apple.com/en-gb/guide/automator/ensm2685/mac).

## Synthesis: dominant pattern

Across the surveyed tools, the dominant pattern is **one memorable top-level container noun plus lower-level building blocks**, not three peer user-facing concepts. The container noun varies by lineage: **Shortcut** (Apple Shortcuts), **Workflow** (Alfred/n8n/Automator), **Zap** (Zapier), **Scenario** (Make), **Applet** (IFTTT), **Macro** (Keyboard Maestro), **Flow** (Power Automate). Tasker is the clearest exception because it exposes **Profiles** and **Tasks** as separate durable primitives, with **Actions** inside tasks. Raycast is another exception, but for a different reason: it is a launcher/AI surface with several object families, and its **Skills** are reusable AI context rather than executable automations. The atomic verb tier is usually called **action** in consumer/no-code products, but graph-oriented tools prefer **node** (n8n) or **module** (Make), and launcher tools prefer **command/object/action panel** language. Triggered vs manual execution is usually represented as a tab, flow type, trigger node/module, or trigger category inside the same top-level container; constrained authoring and advanced authoring usually share the same underlying artifact, with templates, Copilot/AI builders, scripts/code nodes, filters/paths, subflows/subscenarios, or full canvases as the escape hatch.
